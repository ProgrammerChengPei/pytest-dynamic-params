# 需求分析

## 1. 问题分析

在现有的 pytest 测试框架中，当需要结合静态参数（通过 `@pytest.mark.parametrize` 定义）和动态生成的参数时，会遇到以下核心问题：

1. **动态数据生成受限**：
`@pytest.mark.parametrize` 在**模块导入阶段**执行，而 fixture 在**测试运行阶段**才实例化，导致无法在参数化中使用 fixture 的值，限制了动态数据生成的能力

2. **参数组合复杂**：
需要手动编写钩子函数处理参数收集、组合和动态生成，增加了代码复杂度

3. **依赖关系管理困难**：
无法自动处理参数间的依赖关系，需要手动排序，容易出错

4. **嵌套关系支持不足**：
传统pytest对静态参数、动态参数、静态fixture、动态fixture之间的嵌套关系支持有限，无法满足复杂测试场景的需求

## 2. 类型分析

### 2.1 参数类型

| 类型 | 定义 | 特性 | 执行时机 | 来源 | 示例 |
|------|------|------|----------|------|------|
| 静态参数 | 通过 `@pytest.mark.parametrize` 定义的固定参数值 | 值固定，不随测试参数变化 | 模块导入阶段 | pytest 内置特性 | `@pytest.mark.parametrize("input_value", [1, 2, 3])` |
| 动态参数 | 通过 `@param_generator` 定义的运行时生成的参数 | 值可动态生成，支持依赖其他参数或 fixture | 测试运行阶段 | 本项目特性 | `@param_generator def calculate_result(input_value): return input_value * 2` |

### 2.2 Fixture 类型

| 类型 | 定义 | 特性 | 执行时机 | 来源 | 示例 |
|------|------|------|----------|------|------|
| 静态 fixture | 普通的 `@pytest.fixture` 定义的 fixture | 值固定，不随测试参数变化 | 测试运行阶段，在测试函数执行前 | pytest 内置特性 | `@pytest.fixture def database(): return {"users": {"admin": "admin"}}` |
| 动态 fixture | 通过 `@pytest.fixture(params=...)` 定义的参数化 fixture | 会为每个参数值生成不同的 fixture 实例 | 测试运行阶段，在测试函数执行前 | pytest 内置特性 | `@pytest.fixture(params=["dev", "test", "prod"]) def environment(request): return request.param` |

### 2.3 嵌套关系分析

#### 2.3.1 代码示例

以下是模拟用户真实使用场景的代码示例，展示了参数和 fixture 之间的嵌套关系：

```python
# 静态 fixture: 数据库配置
@pytest.fixture
def db_config():
    return {"host": "localhost", "port": 5432}

# 静态 fixture: 应用基础配置
@pytest.fixture
def base_app_config(db_config):
    return {"app_name": "test_app", "version": "1.0.0"}

# 动态 fixture: 环境配置
@pytest.fixture(params=["development", "staging", "production"])
def environment(request):
    return request.param

# 动态 fixture: 依赖环境配置的应用配置
@pytest.fixture
def app_config(environment, db_config):
    return {
        "env": environment,
        "db": db_config,
        "debug": environment == "development"
    }

# 动态参数生成器：基于输入值计算结果
@param_generator
def calculate_result(input_value):
    return input_value * 2

# 动态参数生成器：基于环境生成配置
@param_generator
def generate_env_config(environment):
    return f"config_{environment}"

# 动态参数生成器：基于基础应用配置生成测试数据（使用静态 fixture）
@param_generator
def generate_test_data(base_app_config):
    return {
        "app_name": base_app_config["app_name"],
        "version": base_app_config["version"],
        "test_id": f"test_{base_app_config['app_name']}_1"
    }

# 动态参数生成器：依赖其他动态参数
@param_generator
def generate_derived_value(calculate_result):
    return calculate_result * 10

# 静态 fixture: 使用静态参数 input_value
@pytest.fixture
@pytest.mark.parametrize("input_value", [1, 2, 3])
def static_param_fixture(input_value):
    return input_value * 2

# 静态 fixture: 使用动态参数 calculate_result
@pytest.fixture
@pytest.mark.parametrize("input_value", [1, 2, 3])
def dynamic_param_fixture(input_value, calculate_result):
    return calculate_result == input_value * 2
```

## 3. 需求确定

基于以上问题分析，我们确定了以下核心需求：

1. **动态参数生成**：
   - 在测试运行阶段执行参数生成，解决时机冲突问题
   - 支持依赖fixture的动态数据生成

2. **参数收集与组合**：
   - 自动收集和组合来自不同来源的参数
   - 简化参数管理，减少代码复杂度

3. **依赖关系解析**：
   - 自动分析参数间的依赖关系
   - 按正确顺序执行生成器

4. **嵌套关系支持**：
   - 支持静态参数、动态参数、静态fixture、动态fixture之间的各种嵌套调用关系
   - 满足复杂测试场景的需求

5. **性能优化**：
   - 支持缓存机制，避免重复计算
   - 支持懒加载，减少不必要的计算

6. **与现有机制兼容**：
   - 与现有的 `@pytest.mark.parametrize` 装饰器兼容
   - 与现有的 fixture 系统兼容
   - 与现有的 pytest 钩子和插件兼容

## 4. 应用场景

该动态参数化系统适用于以下场景：

1. **需要在参数化中使用 fixture 值的场景**：如基于数据库连接、环境配置等动态生成测试参数
2. **复杂的依赖链场景**：如参数依赖其他参数，参数依赖 fixture，fixture 依赖参数等多层依赖
3. **计算密集型的参数生成**：如需要大量计算或网络请求生成参数的场景
4. **不同生命周期的参数需求**：如需要在 session 级别缓存的参数，或 function 级别动态生成的参数

## 5. 可行性分析

通过分析现有 pytest 框架的机制和限制，我们认为实现该动态参数化系统是可行的：

1. **技术可行性**：
   - 利用 pytest 的钩子机制，特别是 `pytest_generate_tests` 钩子
   - 利用 fixture 系统的执行时机特性
   - 实现依赖解析算法，处理参数间的依赖关系

2. **实现可行性**：
   - 采用装饰器模式定义参数生成器
   - 实现依赖解析和参数组合逻辑
   - 与现有 pytest 机制无缝集成

3. **兼容性可行性**：
   - 保持与现有 pytest 语法和语义的兼容性
   - 支持现有的测试代码无需修改即可使用新功能

## 6. 需求确认

基于以上分析，我们确认该动态参数化系统的需求是明确的、可行的，并且能够解决传统 pytest 参数化中的核心问题。该系统将为测试工程师提供更强大、更灵活的测试参数化工具，满足复杂测试场景的需求。
