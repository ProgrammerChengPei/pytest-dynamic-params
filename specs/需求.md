# pytest 动态参数化系统需求文档

## 1. 项目概述

### 1.1 背景与问题
在现有的pytest测试框架中，测试用例的参数化主要依赖于`@pytest.mark.parametrize`装饰器和fixture系统。然而，这两种机制在组合使用时存在以下局限性：

1. **参数化时机冲突**：`@pytest.mark.parametrize`在模块导入阶段执行，而fixture在测试运行阶段才实例化
2. **动态数据生成困难**：无法在装饰器参数中调用依赖fixture的函数
3. **复杂的参数组合**：需要手动编写钩子函数处理参数收集、组合和动态生成
4. **依赖管理缺失**：无法自动处理参数间的依赖关系，需要手动排序

### 1.2 解决方案
设计一个通用的动态参数化系统，允许测试开发者声明式地定义参数生成器，系统自动处理：
- 参数收集与组合
- 依赖关系解析
- 动态参数生成
- 测试用例参数化

## 2. 核心设计理念

### 2.1 声明式参数生成
采用声明式API，让测试开发者专注于"生成什么"，而不是"如何生成"：

```python
# 定义参数生成器
@param_generator(scope="module", cache=True)
def generate_user_data(user_id, environment):
    """生成用户数据"""
    return {"id": user_id, "name": f"User_{user_id}"}

# 在测试用例中指定使用
@with_dynamic_params(user_data=generate_user_data)
@pytest.mark.parametrize("user_id", [1, 2, 3])
def test_user_profile(user_id, environment, user_data):
    assert user_data["id"] == user_id
```

### 2.2 自动依赖管理
系统自动分析生成器函数的参数签名，确定其依赖的其他参数（来自`@pytest.mark.parametrize`、fixture或其他生成器），并按正确顺序执行。

### 2.3 与现有机制兼容
系统与现有的pytest机制完全兼容：
- 支持现有的`@pytest.mark.parametrize`装饰器
- 支持现有的fixture系统
- 支持现有的pytest钩子和插件

## 3. 系统架构

### 3.1 组件概述
```
pytest框架
├── 动态参数化插件
│   ├── 参数生成器注册表
│   ├── 依赖解析器
│   ├── 参数组合引擎
│   ├── 作用域管理器
│   └── 缓存管理器
│
├── 装饰器接口
│   ├── @param_generator
│   └── @with_dynamic_params
│
└── 集成点
    ├── pytest_generate_tests钩子
    ├── fixture系统
    └── 测试收集机制
```

### 3.2 执行流程
```
1. 测试收集阶段
   ├── 扫描测试函数，识别@with_dynamic_params装饰器
   ├── 注册所有参数生成器
   ├── 解析每个动态参数的依赖关系
   ├── 确定参数组合总数
   └── 准备测试用例列表

2. 参数生成阶段
   ├── 按依赖顺序执行参数生成器
   ├── 根据作用域缓存生成结果
   ├── 生成所有参数组合
   └── 调用metafunc.parametrize进行参数化

3. 测试执行阶段
   ├── pytest为每个参数组合创建测试实例
   ├── 执行测试函数
   └── 收集测试结果

4. 清理阶段
   └── 根据作用域清理生成器资源
```

## 4. 核心功能需求

### 4.1 参数生成器定义
使用`@param_generator`装饰器定义参数生成器函数：

```python
@param_generator(
    scope: str = "function",  # 作用域：function/class/module/session
    cache: bool = True,       # 是否缓存结果
    name: Optional[str] = None  # 生成器名称（可选）
)
def generator_function(dependency1, dependency2, ...) -> Any:
    """
    参数生成器函数
    
    Args:
        dependency1, dependency2, ...: 依赖的参数或fixture
        
    Returns:
        生成的参数值（单个值或列表）
        
    或者使用yield语法进行资源管理：
        result = generate_data()
        yield result
        # 清理代码
    """
```

**要求**：
- 支持普通函数和生成器函数
- 支持所有Python数据类型作为返回值
- 自动从测试上下文中获取依赖参数

### 4.2 测试用例标记
使用`@with_dynamic_params`装饰器标记使用动态参数的测试用例：

```python
@with_dynamic_params(
    param1=generator_func1,  # param1由generator_func1生成
    param2=generator_func2,  # param2由generator_func2生成
    ...
)
def test_function(static_param1, param1, param2, ...):
    """测试函数可以同时使用静态参数和动态参数"""
```

**要求**：
- 支持多个动态参数
- 支持与`@pytest.mark.parametrize`混合使用
- 支持与普通fixture混合使用

### 4.3 参数依赖处理
系统自动处理以下依赖场景：

#### 4.3.1 简单依赖
```python
@param_generator
def generate_a():
    return "value_a"

@param_generator
def generate_b(a):  # 依赖generate_a的结果
    return f"b_from_{a}"
```

#### 4.3.2 复杂依赖链
```python
@param_generator
def config(environment):
    return {"env": environment, "timeout": 30}

@param_generator
def test_data(config, data_size):
    return [{"id": i, "config": config} for i in range(data_size)]

@param_generator
def expected_results(test_data, validator):
    return [validate(item, validator) for item in test_data]
```

**要求**：
- 自动解析依赖顺序
- 支持多级依赖链
- 检测和报告循环依赖

### 4.4 作用域管理
支持不同级别的作用域控制：

| 作用域 | 生成时机 | 缓存策略 | 适用场景 |
|--------|----------|----------|----------|
| function | 每个测试函数 | 不缓存 | 每个测试需要独立数据 |
| class | 每个测试类 | 类内缓存 | 类中多个测试共享数据 |
| module | 每个模块 | 模块内缓存 | 模块中所有测试共享数据 |
| session | 整个会话 | 全局缓存 | 会话级别共享数据 |

**要求**：
- 支持所有四种作用域
- 支持作用域嵌套
- 自动清理作用域结束时的资源

### 4.5 参数化fixture支持
动态参数生成器可以依赖参数化fixture，系统应为每个fixture参数值分别生成动态参数：

```python
@pytest.fixture(params=[1, 2, 3])
def number(request):
    return request.param

@param_generator
def double(number, multiplier):
    return number * multiplier

@with_dynamic_params(result=double)
@pytest.mark.parametrize("multiplier", [10, 20])
def test_double(number, multiplier, result):
    assert result == number * multiplier
# 测试用例数量：3个number × 2个multiplier = 6个
```

**要求**：
- 识别参数化fixture
- 为每个fixture参数值调用生成器
- 正确计算测试用例数量

## 7. 错误处理

### 7.1 参数验证
当生成器函数需要的参数在测试上下文中不可用时，系统必须报告清晰的错误信息：

**错误信息必须包含**：
- 缺失的参数名称
- 生成器函数名称
- 生成器函数需要的所有参数列表
- 测试上下文中所有可用的参数列表

**错误示例**：
```
错误：参数 'filename' 在生成器函数 'generate_rte_results' 中需要，但未在测试用例中找到。

生成器函数签名：generate_rte_results(viu, filename, reason)
测试用例可用参数：['viu', 'filename2', 'reason', 'columns']
```

### 7.2 依赖解析
检测并报告参数依赖中的循环引用：

**错误示例**：
```
错误：检测到循环依赖

依赖链：
  generate_a → generate_b → generate_a

请修改生成器函数的依赖关系以消除循环。
```

### 7.3 执行错误
当参数生成器执行失败时，提供详细的错误信息：

**错误信息必须包含**：
- 生成器函数名称
- 异常类型和消息
- 调用参数
- 堆栈跟踪

## 8. 使用示例

### 示例1：基础用法（无fixture）
```python
# 最简单场景：只使用静态参数和动态参数
@param_generator
def calculate_result(input_value):
    return input_value * 2

@with_dynamic_params(result=calculate_result)
@pytest.mark.parametrize("input_value", [1, 2, 3])
def test_basic(input_value, result):
    """测试用例数量：3个"""
    assert result == input_value * 2
```

### 示例2：混合参数（含普通fixture）
```python
# 引入普通fixture
@pytest.fixture
def database():
    db = connect()
    yield db
    db.close()

@param_generator
def get_user_data(database, user_type):
    return database.query_users_by_type(user_type)

@with_dynamic_params(user_data=get_user_data)
@pytest.mark.parametrize("user_type", ["admin", "user"])
def test_with_fixture(database, user_type, user_data):
    """测试用例数量：2个"""
    assert user_data["type"] == user_type
    assert database.is_connected()
```

### 示例3：参数化fixture
```python
# 引入参数化fixture
@pytest.fixture(params=["dev", "test", "prod"])
def environment(request):
    return {"env": request.param, "timeout": 30}

@param_generator
def generate_config(environment, feature_flag):
    return {
        "env": environment["env"],
        "feature": feature_flag,
        "timeout": environment["timeout"] + 10
    }

@with_dynamic_params(config=generate_config)
@pytest.mark.parametrize("feature_flag", ["A/B", "control"])
def test_parametrized_fixture(environment, feature_flag, config):
    """测试用例数量：3个环境 × 2个feature_flag = 6个"""
    assert config["env"] == environment["env"]
    assert config["feature"] == feature_flag
```

### 示例4：多个fixture嵌套
```python
# fixture之间嵌套依赖
@pytest.fixture
def base_config():
    return {"app": "test", "version": "1.0"}

@pytest.fixture
def database_config(base_config):
    return {**base_config, "db": "postgresql"}

@pytest.fixture
def app_config(database_config):
    return {**database_config, "port": 8080}

@param_generator
def generate_test_data(app_config, test_type):
    return {
        "config": app_config,
        "type": test_type,
        "data": [1, 2, 3]
    }

@with_dynamic_params(test_data=generate_test_data)
@pytest.mark.parametrize("test_type", ["unit", "integration"])
def test_fixture_nesting(app_config, test_type, test_data):
    """fixture依赖链：base_config → database_config → app_config → generate_test_data"""
    assert test_data["config"]["port"] == 8080
    assert test_data["type"] == test_type
```

### 示例5：多个动态参数嵌套
```python
# 动态参数之间嵌套依赖
@param_generator
def get_raw_data(data_source, size):
    return [{"id": i, "source": data_source} for i in range(size)]

@param_generator
def process_data(raw_data, algorithm):
    return [apply_algorithm(item, algorithm) for item in raw_data]

@param_generator
def validate_results(processed_data, threshold):
    return all(score >= threshold for score in processed_data)

@with_dynamic_params(
    raw_data=get_raw_data,
    processed_data=process_data,
    is_valid=validate_results
)
@pytest.mark.parametrize("data_source", ["api", "database"])
@pytest.mark.parametrize("size", [5, 10])
@pytest.mark.parametrize("algorithm", ["algo1", "algo2"])
@pytest.mark.parametrize("threshold", [0.5, 0.8])
def test_dynamic_params_nesting(
    data_source, size, algorithm, threshold,
    raw_data, processed_data, is_valid
):
    """动态参数依赖链：data_source/size → raw_data → processed_data → is_valid"""
    assert len(raw_data) == size
    assert len(processed_data) == size
    assert isinstance(is_valid, bool)
```
