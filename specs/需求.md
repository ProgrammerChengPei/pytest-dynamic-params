# pytest 动态参数化系统需求文档

## 1. 项目概述

在现有的pytest测试框架中，测试用例的参数化主要依赖于`@pytest.mark.parametrize`装饰器和fixture系统。然而，这两种机制在组合使用时存在局限性，我们设计了一个通用的动态参数化系统来解决这些问题：

1. **动态数据生成受限**：
   - **问题**：`@pytest.mark.parametrize`在模块导入阶段执行，而fixture在测试运行阶段才实例化，导致无法在参数化中使用fixture的值，限制了动态数据生成的能力
   - **解决方案**：在测试运行阶段执行参数生成，解决时机冲突问题，支持依赖fixture的动态数据生成

2. **参数组合复杂**：
   - **问题**：需要手动编写钩子函数处理参数收集、组合和动态生成，增加了代码复杂度
   - **解决方案**：自动收集和组合来自不同来源的参数，简化参数管理

3. **依赖关系管理困难**：
   - **问题**：无法自动处理参数间的依赖关系，需要手动排序，容易出错
   - **解决方案**：自动分析参数间的依赖关系，按正确顺序执行生成器

4. **嵌套关系支持不足**：
   - **问题**：传统pytest对静态参数、动态参数、静态fixture、动态fixture之间的嵌套关系支持有限，无法满足复杂测试场景的需求
   - **解决方案**：支持静态参数、动态参数、静态fixture、动态fixture之间的各种嵌套调用关系，满足复杂测试场景的需求

## 2. 核心设计理念

### 2.1 声明式参数生成

采用声明式API，让测试开发者专注于"生成什么"，而不是"如何生成"：

```python
# 定义参数生成器
@param_generator(scope="module", cache=True)
def generate_user_data(user_id, environment):
    """生成用户数据"""
    return {"id": user_id, "name": f"User_{user_id}"}

# 在测试用例中指定使用
@dynamic_params(user_data=generate_user_data)
@pytest.mark.parametrize("user_id", [1, 2, 3])
def test_user_profile(user_id, environment, user_data):
    assert user_data["id"] == user_id
```

### 2.2 自动依赖管理

系统自动分析生成器函数的参数签名，确定其依赖的其他参数（来自`@pytest.mark.parametrize`、fixture或其他生成器），并按正确顺序执行。支持复杂的依赖链和嵌套关系。

### 2.3 与现有机制兼容

系统与现有的pytest机制完全兼容：

- 支持现有的`@pytest.mark.parametrize`装饰器
- 支持现有的fixture系统
- 支持现有的pytest钩子和插件

## 3. 系统架构

### 3.1 组件概述

```
pytest框架
├── 动态参数化插件
│   ├── 参数生成器注册表
│   ├── 依赖解析器
│   ├── 参数组合引擎
│   ├── 作用域管理器
│   └── 缓存管理器
│
├── 装饰器接口
│   ├── @param_generator
│   └── @dynamic_params
│
└── 集成点
    ├── pytest_generate_tests钩子
    ├── fixture系统
    └── 测试收集机制
```

### 3.2 执行流程

```
1. 测试收集阶段
   ├── 扫描测试函数，识别@dynamic_params装饰器
   ├── 注册所有参数生成器
   ├── 解析每个动态参数的依赖关系
   ├── 确定参数组合总数
   └── 准备测试用例列表

2. 参数生成阶段
   ├── 按依赖顺序执行参数生成器
   ├── 根据作用域缓存生成结果
   ├── 生成所有参数组合
   └── 调用metafunc.parametrize进行参数化

3. 测试执行阶段
   ├── pytest为每个参数组合创建测试实例
   ├── 执行测试函数
   └── 收集测试结果

4. 清理阶段
   └── 根据作用域清理生成器资源
```

## 4. 核心功能需求

### 4.1 参数生成器定义

使用`@param_generator`装饰器定义参数生成器函数：

```python
@param_generator(
    scope: str = "function",  # 作用域：function/class/module/session
    cache: bool = True,       # 是否缓存结果
    name: Optional[str] = None  # 生成器名称（可选）
)
def generator_function(dependency1, dependency2, ...) -> Any:
    """
    参数生成器函数
    
    Args:
        dependency1, dependency2, ...: 依赖的参数或fixture
        
    Returns:
        生成的参数值（单个值或列表）
        
    或者使用yield语法进行资源管理：
        result = generate_data()
        yield result
        # 清理代码
    """
```

**要求**：

- 支持普通函数和生成器函数
- 支持所有Python数据类型作为返回值
- 自动从测试上下文中获取依赖参数

### 4.2 测试用例标记

使用`@dynamic_params`装饰器标记使用动态参数的测试用例：

```python
@dynamic_params(
    param1=generator_func1,  # param1由generator_func1生成
    param2=generator_func2,  # param2由generator_func2生成
    ...
)
def test_function(static_param1, param1, param2, ...):
    """测试函数可以同时使用静态参数和动态参数"""
```

**要求**：

- 支持多个动态参数
- 支持与`@pytest.mark.parametrize`混合使用
- 支持与普通fixture混合使用

### 4.3 参数依赖处理

系统自动处理以下依赖场景：

#### 4.3.1 简单依赖

```python
@param_generator
def generate_a():
    return "value_a"

@param_generator
def generate_b(a):  # 依赖generate_a的结果
    return f"b_from_{a}"
```

#### 4.3.2 复杂依赖链

```python
@param_generator
def config(environment):
    return {"env": environment, "timeout": 30}

@param_generator
def test_data(config, data_size):
    return [{"id": i, "config": config} for i in range(data_size)]

@param_generator
def expected_results(test_data, validator):
    return [validate(item, validator) for item in test_data]
```

**要求**：

- 自动解析依赖顺序
- 支持多级依赖链
- 检测和报告循环依赖

### 4.4 作用域管理

支持不同级别的作用域控制：

| 作用域      | 生成时机   | 缓存策略  | 适用场景        |
| -------- | ------ | ----- | ----------- |
| function | 每个测试函数 | 不缓存   | 每个测试需要独立数据  |
| class    | 每个测试类  | 类内缓存  | 类中多个测试共享数据  |
| module   | 每个模块   | 模块内缓存 | 模块中所有测试共享数据 |
| session  | 整个会话   | 全局缓存  | 会话级别共享数据    |

**要求**：

- 支持所有四种作用域
- 支持作用域嵌套
- 自动清理作用域结束时的资源

### 4.5 嵌套关系支持

在介绍嵌套关系之前，先明确以下概念：

**参数类型**：

| 类型 | 定义 | 特性 | 执行时机 | 来源 | 示例 |
|------|------|------|----------|------|------|
| 静态参数 | 通过 `@pytest.mark.parametrize` 定义的固定参数值 | 值固定，不随测试参数变化 | 模块导入阶段 | pytest 内置特性 | `@pytest.mark.parametrize("input_value", [1, 2, 3])` |
| 动态参数 | 通过 `@param_generator` 定义的运行时生成的参数 | 值可动态生成，支持依赖其他参数或 fixture | 测试运行阶段 | 本项目特性 | `@param_generator def calculate_result(input_value): return input_value * 2` |

**Fixture 类型**：

| 类型 | 定义 | 特性 | 执行时机 | 来源 | 示例 |
|------|------|------|----------|------|------|
| 静态 fixture | 普通的 `@pytest.fixture` 定义的 fixture | 值固定，不随测试参数变化 | 测试运行阶段，在测试函数执行前 | pytest 内置特性 | `@pytest.fixture def database(): return {"users": {"admin": "admin"}}` |
| 动态 fixture | 通过 `@pytest.fixture(params=...)` 定义的参数化 fixture | 会为每个参数值生成不同的 fixture 实例 | 测试运行阶段，在测试函数执行前 | pytest 内置特性 | `@pytest.fixture(params=["dev", "test", "prod"]) def environment(request): return request.param` |

支持各种参数和fixture之间的嵌套调用关系，包括：

2. **静态参数和动态参数混合使用**
   - 描述：？
   - 执行时机：测试运行阶段

   - 示例：
        - 示例 1：动态参数直接调用静态参数

            ```python
            @pytest.mark.parametrize("input_value", [1, 2, 3])
            ```

        - 示例 2: 测试用例混合使用静态参数和动态参数

6. **动态参数调用动态参数**
   - 描述：动态参数之间的依赖
   - 执行时机：测试运行阶段
   - 实现方式：系统自动分析依赖关系，按正确顺序执行
   - 示例：
     ```python
     @param_generator
     def calculate_result(input_value):
         return input_value * 2
     
     @param_generator
     def generate_derived_value(calculate_result):
         return calculate_result * 10
     
     @pytest.mark.parametrize("input_value", [1, 2])
     @dynamic_params(result=calculate_result, derived=generate_derived_value)
     def test_dynamic_param_calls_dynamic_param(input_value, result, derived):
         assert result == input_value * 2
         assert derived == result * 10
     ```

## 6. 配置

### 6.1 配置文件

```ini
# pytest.ini
[pytest]
dynamic_param_cache_enabled = true
dynamic_param_validation = strict
dynamic_param_log_level = INFO
```

## 7. 错误处理

### 7.1 参数验证

当生成器函数需要的参数在测试上下文中不可用时，系统必须报告清晰的错误信息：

**错误信息必须包含**：

- 缺失的参数名称
- 生成器函数名称
- 生成器函数需要的所有参数列表
- 测试上下文中所有可用的参数列表

**错误示例**：

```
错误：参数 'filename' 在生成器函数 'generate_rte_results' 中需要，但未在测试用例中找到。

生成器函数签名：generate_rte_results(viu, filename, reason)
测试用例可用参数：['viu', 'filename2', 'reason', 'columns']
```

### 7.2 依赖解析

检测并报告参数依赖中的循环引用：

**错误示例**：

```
错误：检测到循环依赖

依赖链：
  generate_a → generate_b → generate_a

请修改生成器函数的依赖关系以消除循环。
```

### 7.3 执行错误

当参数生成器执行失败时，提供详细的错误信息：

**错误信息必须包含**：

- 生成器函数名称
- 异常类型和消息
- 调用参数
- 堆栈跟踪
